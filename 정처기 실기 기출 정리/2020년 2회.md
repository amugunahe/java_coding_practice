# 문제 1

**문제**  
일정한 주기를 가지고 계속 검토해 나가며 필요할 때마다 요구사항을 더하고 수정하여 커다랗게 살을 붙이면서 개발해 나가는 프로세스 모델 방식이다.  
미리 정해진 몇 개의 단계에 따라 엄격한 순서대로 이루어지는 일직선의 과정인 폭포수의 프로세스와는 비교가 많이 되는 반대의 개념이다.  
이 소프트웨어 개발 방법론을 쓰시오.

**답**  
애자일

---

## 생명주기 모형의 종류

### 1. 폭포수 모형  
- 단계: 계획 → 요구사항 분석 → 설계 → 구현 → 테스트 → 유지보수  
- 특징: 각 단계가 순차적으로 진행되며, 엄격한 순서를 따름

### 2. 프로토타입 모형  
- 설명: 실제 개발될 시스템의 견본을 미리 만들어 최종 결과물을 예측하는 모형

### 3. 나선형 모형  
- 단계: 계획 수립 → 위험분석 → 공학적 개발 → 고객 평가  
- 특징: 위험 관리에 중점을 둔 반복적 개발 모형

### 4. 애자일 방법론  
- XP (익스트림 프로그래밍)  
  - 핵심 가치: 의사소통, 단순성, 피드백, 용기, 존중  
  - 특징: 고객과 함께 2주 정도 반복 개발, 테스트와 우선 개발 강조  

- Scrum  
  - 프로젝트를 30일마다 동작 가능한 제품을 제공하는 스프린트로 분리  
  - 매일 약 15분 회의 실시 (데일리 스크럼)  

- 특징 주도 개발 (FDD)  
  - 2주 정도의 반복 개발 실시  

- 크리스털  
  - 애자일 방법론 중 하나로, 프로젝트 특성에 맞게 유연하게 적용하는 방식

# 문제 4

**문제**  
( )는 소프트웨어 변경사항을 체계적으로 추적하고 통제하는 것으로, ( )는 일반적인 단순 버전 관리 기반의 소프트웨어 운용을 좀 더 포괄적인 학술 분야의 형태로 넓히는 개념을 말한다.  
( )란 소프트웨어 소스코드 뿐만 아니라 개발 환경, 빌드 구조 등 전반적인 환경 내역에 대한 관리체계를 정의하고 있다.  
이를 지원하는 도구로 Git, SVN, CVS 등이 있다.

**답**  
형상관리, Software Configuration Management (SCM)

---

## 설명  
- **형상관리**  
  소프트웨어 변경사항을 체계적으로 추적하고 통제하는 활동

- **Software Configuration Management (SCM)**  
  단순 버전 관리를 넘어서 개발 환경, 빌드 구조 등 소프트웨어 개발과 관련된 모든 환경을 포괄적으로 관리하는 학술적 개념

- **지원 도구**  
  Git, SVN, CVS 등 버전 관리 및 형상관리를 지원하는 도구들


# 관련 개념: 형상관리 및 연관 용어

## 변경관리 (Change Management)
- 변경된 내용과 이력 관리를 체계적으로 수행하여 개발 과정에서 발생할 수 있는 혼란과 오류를 줄임.

## 버전관리 (Version Control)
- 파일의 변경 이력을 관리하며, 필요 시 특정 시점으로 복원하거나 변경 내용 비교 가능.

## 형상관리 (Configuration Management)
- 단순 소스 코드 관리뿐 아니라 빌드 환경, 개발 도구 설정, 문서, 테스트 케이스 등  
  소프트웨어 개발에 관련된 모든 요소를 포괄적으로 관리.  
- 이를 통해 소프트웨어 품질 보증과 일관된 배포가 가능.

---

## 형상관리 절차

### 1. 형상 식별 (Configuration Identification)
- 관리 대상인 소프트웨어 구성 요소 및 관련 산출물을 명확히 정의

### 2. 형상 통제 (Configuration Control)
- 변경 요청 및 승인 과정을 통해 변경사항을 통제하며, 무분별한 변경 방지

### 3. 형상 감사 (Configuration Audit)
- 관리 대상의 실제 상태가 문서화된 상태와 일치하는지 검증

### 4. 형상 기록 (Configuration Status Accounting)
- 변경 이력과 상태를 기록하고 보고하여 추적 가능하도록 함

---

## 버전관리 도구 분류 및 특징

### 1. 공유 폴더 방식 (SCCS, RCS, PVCS)
- 파일이 저장된 공유 폴더에 직접 접근하여 버전 관리 수행  
- 단순한 소규모 프로젝트에 적합

### 2. 클라이언트/서버 방식 (CVS, SVN, ClearCase)
- 중앙 서버에 버전 저장소를 두고, 클라이언트가 서버와 통신하며 버전 관리  
- 협업과 대규모 프로젝트에 적합

### 3. 분산 저장소 방식 (Git, GNU arch, Bazaar, Bitkeeper)
- 각 개발자가 전체 저장소를 로컬에 복제하여 독립적으로 작업 가능  
- 네트워크 연결이 없더라도 작업 가능하며, 병합과 분기가 유연함

---

## 추가 설명

### 형상관리의 중요성
- 프로젝트의 복잡도가 증가함에 따라 여러 구성 요소와 다양한 버전 관리가 필수적임  
- 팀원 간의 협업을 원활하게 하고, 오류 발생 시 신속한 복구와 원인 분석을 가능하게 함  
- 제품 릴리즈 시 일관된 빌드와 배포를 보장하여 품질 유지에 기여

### 형상관리 도구 선택 기준
- 프로젝트 규모 및 협업 인원 수  
- 분산 작업 필요성 여부  
- 기존 개발 프로세스와의 호환성  
- 사용 편의성과 커뮤니티 지원

# 문제 5

**문제**  
웹페이지 전체를 다시 로딩하지 않고도, 웹페이지의 일부만을 갱신할 수 있는 비동기 방식의 처리 기술 약어를 쓰시오.

**답**  
AJAX

---

## 관련 설명

### AJAX (Asynchronous JavaScript and XML)
- 웹페이지에서 전체를 다시 불러오지 않고 필요한 부분만 비동기적으로 서버와 데이터를 주고받아 갱신하는 기술  
- 사용자 경험(UX)을 향상시키고, 서버 부하를 줄이며, 빠른 인터랙티브 웹 애플리케이션 구현에 사용됨  
- XML뿐만 아니라 JSON, HTML, 텍스트 등 다양한 데이터 형식을 사용할 수 있음  

### 동작 원리
1. 자바스크립트가 XMLHttpRequest 객체를 생성  
2. 비동기적으로 서버에 요청 전송  
3. 서버에서 응답을 받아 웹페이지의 일부를 동적으로 변경  

### 참고  
- AJAX는 단독 기술이 아니라 여러 기술의 조합(JavaScript, DOM, XMLHttpRequest, HTML, CSS 등)으로 구성됨

# 관련 설명: UI 설계 원칙

## 직관성 (Intuitiveness)
- 사용자가 별도의 교육 없이도 자연스럽게 시스템을 이해하고 사용할 수 있어야 함.

- UI가 사용자가 의도한 목표를 정확하게 지원하고, 올바른 작업을 수행할 수 있게 해야 함.

## 학습성 (Learnability)
- 처음 사용하는 사용자도 쉽게 배울 수 있어야 하며, 반복 사용 시 익숙해져야 함.

## 유연성 (Flexibility)
- 다양한 사용자의 요구를 수용하고, 실수 발생 시 오류를 최소화하거나 쉽게 복구할 수 있어야 함.

# 문제 7

**문제**  
프로그램을 실행시키지 않고, 소스코드에 대한 코딩 표준, 코딩 스타일, 복잡도, 보안 취약점, 잠재적인 결함을 찾는 테스트 활동은?

**답**  
정적분석, 정적테스트


# 테스트 유형별 분류

## 1. 단위 테스트 (Unit Testing)
- **정적 테스트**: 코드 실행 없이 코드 자체를 분석  
- **동적 테스트**: 실제로 프로그램을 실행하여 테스트

## 2. 통합 테스트 (Integration Testing)
- **상향 테스트 (Bottom-up)**: 하위 모듈부터 테스트하며 상위 모듈로 진행, 드라이버 사용  
- **하향 테스트 (Top-down)**: 상위 모듈부터 테스트하며 하위 모듈로 진행, 스텁 사용  
- **빅뱅 테스트**: 모든 모듈을 한꺼번에 통합하여 테스트

## 3. 시스템 테스트 (System Testing)
- **기능 테스트**: 버튼 클릭 등 기능이 제대로 동작하는지 확인  
- **비기능 테스트**: 안정성, 성능, 보안 등 비기능적 요구사항 평가

## 4. 인수 테스트 (Acceptance Testing)
- **알파 테스트**: 개발자와 사용자가 함께 수행하는 테스트  
- **베타 테스트**: 실제 사용자가 수행하며 개발자는 관여하지 않음

---

# 어플리케이션 테스트 유형 분류

| 분류         | 내용                                        |
|--------------|-------------------------------------------|
| 프로그램 실행 여부 | 정적 테스트 / 동적 테스트                     |
| 테스트 기법      | 화이트박스 테스트 / 블랙박스 테스트            |
| 테스트에 대한 시각 | 검증 테스트(Verification) / 확인 테스트(Validation) |
| 테스트 목적      | 회복, 안전, 강도, 성능, 구조, 회귀, 병행 테스트 등    |
| 테스트 기반      | 명세 기반, 구조 기반, 경험 기반 테스트              |

---

## 추가 설명

### 정적분석 (Static Analysis)
- 프로그램을 실행하지 않고 코드의 품질, 보안, 스타일, 복잡도 등을 분석하여 잠재적 결함을 발견하는 기법  
- 도구 예: SonarQube, FindBugs, PMD 등

### 동적분석 (Dynamic Analysis)
- 실제로 프로그램을 실행하여 기능과 성능을 검증하는 테스트

### 화이트박스 테스트 (White-box Testing)
- 내부 로직을 알고 테스트하는 방식 (코드 커버리지 중심)

### 블랙박스 테스트 (Black-box Testing)
- 내부 구조를 모른 채 외부 명세에 따라 테스트하는 방식

### 검증(Verification)
- 올바르게 만들어졌는지 점검하는 활동

### 확인(Validation)
- 사용자의 요구사항에 맞는지 점검하는 활동

# 문제 11] SQL 제어어(DCL) 중 ROLLBACK 명령에 대해 간략히 설명하시오

**답:**  
데이터베이스 내의 연산이 비정상적으로 종료되거나, 정상 수행이 되었더라도 이전 상태로 되돌리기 위한 명령어이다.  
즉, 트랜잭션 수행 중 오류 발생 시 **수행 전 상태로 복구**할 수 있도록 한다.

---

## 관련 개념 요약

### ✅ SQL 명령어 종류

- **DDL (데이터 정의어)**  
  - `CREATE`, `ALTER`, `DROP`, `TRUNCATE`  
  - 테이블, 뷰, 인덱스 등 구조를 정의하거나 변경

- **DML (데이터 조작어)**  
  - `SELECT`, `INSERT`, `UPDATE`, `DELETE`  
  - 데이터를 조회하거나 조작

- **DCL (데이터 제어어)**  
  - `GRANT`, `REVOKE`: 사용자에게 권한 부여/회수  
  - `COMMIT`: 트랜잭션에서 수행한 변경사항을 **영구 반영**  
  - `ROLLBACK`: 트랜잭션에서 수행한 변경사항을 **취소 및 이전 상태로 복구**

---

## 트랜잭션의 4가지 특성 (ACID)

| 특성     | 설명 |
|----------|------|
| 원자성 (Atomicity) | 모든 작업이 전부 수행되거나, 전혀 수행되지 않아야 함 (`COMMIT` 또는 `ROLLBACK`) |
| 일관성 (Consistency) | 트랜잭션 수행 전과 후에 DB의 상태가 일관되어야 함 |
| 격리성 (Isolation) | 동시에 수행되는 트랜잭션들이 서로 영향을 미치지 않도록 보장 |
| 영속성 (Durability) | 트랜잭션이 성공적으로 완료되면, 결과는 영구적으로 반영되어야 함 |

# 문제 12] 네트워크 계층(Network Layer, 3계층)에서 IP 패킷 단위의 데이터 변조 방지 및 암호화 기능을 제공하는 통신 규약을 쓰시오

**답:**  
IPSec

---

## 📌 관련 개념 정리

### ✅ OSI 7계층 요약

| 계층 | 주요 기능 | 예시 |
|------|-----------|------|
| 7. 응용(Application) | 사용자 인터페이스, 이메일, 웹 | HTTP, FTP |
| 6. 표현(Presentation) | 데이터 표현, 암호화/복호화 | JPEG, ASCII |
| 5. 세션(Session) | 연결 관리, 동기화 | NetBIOS |
| 4. 전송(Transport) | 종단 간 신뢰성 있는 데이터 전송 | TCP, UDP |
| 3. 네트워크(Network) | 경로 선택, 주소 지정 | IP, ICMP |
| 2. 데이터링크(Data Link) | 오류 검출, 프레임 전송 | MAC, Switch |
| 1. 물리(Physical) | 비트 전송 | 케이블, 허브 |

---

### ✅ IPSec (IP Security Protocol)

- **계층 위치:** 네트워크 계층 (3계층)
- **기능:**  
  - IP 패킷의 **암호화**  
  - 데이터 **기밀성, 무결성, 인증** 제공  
  - **종단 간 보안** 또는 **게이트웨이 간 보안** 가능

---

### IPSec 구성 프로토콜

| 구성 요소 | 설명 |
|-----------|------|
| **AH (Authentication Header)** | IP 패킷의 **무결성**과 **인증**을 제공. 변조 방지 |
| **ESP (Encapsulating Security Payload)** | IP 페이로드의 **암호화**를 통해 **기밀성** 제공 |

---

### ✅ 비교: 보안 관련 프로토콜

| 프로토콜 | 사용 계층 | 특징 |
|----------|-----------|------|
| **IPSec** | 네트워크 계층 | IP 패킷 단위 보안, VPN 등에서 사용 |
| **SSL / TLS** | 전송 계층 위 | HTTPS, 브라우저 보안 통신 |
| **S-HTTP** | 응용 계층 | HTTP에 보안 기능 추가 |

# 문제 15

**문제**  
다음은 IP 인프라 서비스 관리 실무와 관련된 <실무 사례>에 대한 설명이다. 적합한 용어를 한글 또는 영문으로 써라.

---

## <실무 사례>  
A는 (주)한국 아이티 보안관제실에서 정보시스템의 정보관리를 위해 모니터링을 담당하며 근무하고 있다.  
시스템 운영 중 **자연재해나 시스템 장애** 등의 이유로 **대고객 서비스가 불가능한 경우**가 종종 발생한다.  
A는 이런 **장애 발생 시 서비스를 정상적으로 수행하는 구축 시스템을 관리**하는 역할을 한다.  
이 용어는 이런 **비상 상황이 발생했을 때 '서비스 중단 시점과 서비스 복원 시점 간의 최대 지연 시간'**을 가리킨다.

---

**답**  
복구 시간 목표 (Recovery Time Objective, RTO)

---

## 📌 관련 개념 정리

### ✅ RTO (Recovery Time Objective) - 목표 복구 시간
- 시스템 장애 발생 **이후부터 복구까지 허용 가능한 최대 시간**
- 예: RTO가 2시간이면, **2시간 이내에 시스템이 복구되어야 함**

### ✅ RPO (Recovery Point Objective) - 목표 복구 시점
- 장애 발생 시점 기준으로 **데이터 손실을 허용할 수 있는 시점**
- 예: RPO가 10분이면, 마지막 백업 시점이 **10분 전까지 보장**되어야 함

---

### ✅ 백업 유형

| 유형       | 설명 |
|------------|------|
| 전체 백업   | 모든 데이터를 백업하는 방식. 가장 안정적이지만 **시간과 용량** 많이 소모됨 |
| 증분 백업   | **이전 백업 이후 변경된 데이터만** 백업. 빠르고 용량 적지만 복구 시 순차적으로 필요 |
| 차등 백업   | **전체 백업 이후 변경된 모든 데이터** 백업. 복구 시 전체 + 마지막 차등만 필요 |

자동으로 내용이 갱신되는 방식으로 일대다(one to many) 의존성을 정의하는 디자인 패턴명을 **영문으로 쓰시오**  

---

**답:**

---

## 📌 관련 개념 정리

### ✅ Observer Pattern (옵서버 패턴)

| 항목 | 설명 |
|------|------|
| 정의 | **객체의 상태 변화**를 감지하여, 그 상태에 **의존하는 객체들에 자동으로 알림**을 보내는 디자인 패턴 |

- Subject: 상태를 가지고 있으며, 관찰자를 등록/제거/알림 처리  
- Observer: Subject에 의존하여 상태를 갱신 받음 |
| 특징 | - **일대다(one-to-many)** 관계  
- **느슨한 결합** 유지  
- 상태 변경을 자동 전파 |
| 활용 예시 | - GUI 이벤트 리스너  
- 뉴스 구독 시스템  
- 모델-뷰-컨트롤러(MVC) 패턴의 View 갱신 방식 |

# 문제 17]  
다음 설명에 해당하는 **용어를 쓰시오**


- 자유롭게 사용이 가능하며, 저작권 표시 및 변경 허락 조건하에 **재배포 가능한 링크 기반 개방형 데이터**  
- **연계 데이터**와 **오픈 데이터**가 결합된 단어  
- 웹에 게시되는 데이터에 **식별자(URI)**를 부여하고 관련 정보를 구조적으로 제공  
- 저작권 없이 **무료로 제공**, 사용자가 정보를 **다양하고 효율적으로 활용**할 수 있도록 함  
- **HTTP, RDF, URI** 등의 웹표준 기술을 기반으로 시멘틱 웹을 구성  
- 출처가 다른 데이터라도 **URI를 통해 서로 연결**하여 활용  
- **데이터 재사용 가능**, **중복 감소**라는 장점이 있음

---

**답:**  
**Linked Open Data (LOD)**

---

## 📌 관련 개념 설명

### ✅ Linked Open Data (LOD)

| 항목 | 내용 |
|------|------|
| 정의 | **웹 상에 공개된 데이터**를 의미하며, 각 데이터에 **고유 식별자(URI)**를 부여하고, 다른 데이터와의 **관계(링크)**를 표현하여 **시멘틱 웹(Semantic Web)**을 실현하기 위한 기술 |
| 핵심 기술 | - **HTTP** (자원 요청 및 접근)  
- **URI** (식별자 부여)  
- **RDF** (자원 간 관계 표현) |
| 장점 | - 데이터 재사용 가능  
- 데이터 간 연결성 증가  
- 시멘틱 웹 기반의 자동화된 데이터 활용 가능 |
| 활용 예시 | - 국가 데이터 포털  
- 도서관의 서지 정보  
- 공공기관의 행정 데이터 개방 등 |

---

### ✅ 시멘틱 웹 (Semantic Web)

- 의미를 이해할 수 있도록 구조화된 웹  
- 컴퓨터가 데이터를 **자동으로 이해 및 추론**할 수 있는 **차세대 지능형 웹**  
- RDF, OWL, SPARQL 등 **메타데이터 기술**을 활용

---

### ✅ REST (Representational State Transfer)

- 자원을 **URI**로 명확하게 표현하고  
- HTTP 메서드(GET, POST, PUT, DELETE)를 활용하여 **자원에 대한 CRUD 수행**  
- REST의 설계 원칙을 따르는 서비스를 **RESTful** 웹 서비스라고 함
