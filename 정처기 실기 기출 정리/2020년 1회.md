# 문제 1) 비정규화(반정규화, 역정규화) 개념 간략 설명

**답:**  
성능 향상 // 성능 향상과 개발 편의성등 정규화 기법에 위배되는 수행기법

**추가 설명:**  
비정규화는 데이터베이스 설계에서 정규화된 상태에서 성능 향상이나 개발 편의성을 위해 일부러 정규화 규칙을 완화하는 기법입니다. 정규화는 데이터 중복을 줄이고 데이터 무결성을 강화하지만, 조인 연산 증가 등으로 인해 조회 성능 저하가 발생할 수 있습니다.  
따라서 비정규화는 이러한 성능 문제를 해결하기 위해 데이터 중복을 허용하여 조회 속도를 개선하는 방법이며, 이는 정규화 원칙에 위배되지만 실무에서는 성능 최적화를 위해 자주 사용됩니다.

---

## 문제 1 해설

- 개념적 설계: ERD(Entity Relationship Diagram) 작성  
- 논리적 설계: 목표 DBMS 선정, 트랜잭션 인터페이스(T/I), 정규화 수행  
- 물리적 설계: 성능 최적화를 위한 파티션, 클러스터, 인덱스 설계  

### 정규화 단계  
- 1정규형 (1NF): 도메인이 원자값(분해 불가능한 값)이어야 함  
- 2정규형 (2NF): 부분 함수 종속 제거 (기본키의 일부에만 종속된 속성 제거)  
- 3정규형 (3NF): 이행 함수 종속 제거 (기본키가 아닌 속성이 다른 기본키가 아닌 속성에 종속되지 않아야 함)  
- BCNF (보이스-코드 정규형): 결정자이면서 후보키가 아닌 속성 제거  
- 4정규형 (4NF): 다치 종속 제거 (하나의 기본키에 대해 여러 독립적인 다치 종속성 제거)  
- 5정규형 (5NF): 조인 종속 제거 (조인 연산을 통해만 복원 가능한 종속성 제거)

# 문제 2) 마크업 언어

() : SGML의 단순화된 부분 집합, 다른 많은 종류의 데이터를 기술하는데 사용할 수 있다.  
() : 주로 다른 종류 시스템, 인터넷에 연결된 시스템끼리 데이터를 쉽게 주고 받을 수 있게 하여 HTML 한계를 극복할 목적으로 만들어짐  
() : 텍스트 데이터 형식으로 유니코드 사용해 전세계 언어를 지원한다.  

**답:** XML

---

## 문제 2 해설

- **마크업 언어**  
  - HTML: 주로 웹 페이지의 구조와 내용을 담당  
  - SGML: 복잡하여 실제 사용이 제한적임  
  - XML: SGML을 단순화하여 확장된 마크업 언어로, 다양한 데이터를 기술 가능하며 시스템 간 데이터 교환에 적합  
  - JSON: 키와 값이 쌍으로 이루어진 경량 데이터 교환 형식  
  - TXT: 단순한 텍스트 파일 형식  
  - YAML: 직렬화 언어로, 사람에게 읽기 쉬운 데이터 표현 방식

---

### 추가 설명  
- XML(eXtensible Markup Language)은 SGML(Standard Generalized Markup Language)의 단순화된 부분 집합으로, 데이터의 구조와 의미를 표현하기 위해 설계된 텍스트 기반 형식입니다.  
- XML은 시스템 간의 데이터 교환을 용이하게 하고, HTML이 주로 웹페이지 표시 목적이라면 XML은 데이터 기술과 전송에 초점을 둡니다.  
- 유니코드 지원을 통해 전 세계 거의 모든 언어를 표현할 수 있어 국제화에 강점을 가지고 있습니다.

# 문제 3) 살충제 패러독스 개념 간략 설명

**답:** 동일한 테스트 케이스로 동일한 절차를 반복 수행하면 새로운 결함을 찾을 수 없다.

---

## 문제 3 해설

- **소프트웨어 테스트 종류**  
  - **단위 테스트**  
    - 정적 테스트: 소스 코드를 직접 검토  
    - 동적 테스트: 소스 코드를 실행하며 검증  
  - **통합 테스트**  
    - 하향식 테스트: 상위 모듈부터 테스트하며 하위 모듈은 스텁(stub)으로 대체  
    - 상향식 테스트: 하위 모듈부터 테스트하며 상위 모듈은 드라이버(driver)로 대체  
    - 빅뱅 테스트: 모든 모듈을 한꺼번에 통합해 테스트, 점진적이지 않음  
  - **시스템 테스트**  
    - 기능 테스트: 요구 기능 검증  
    - 비기능 테스트: 성능, 보안 등 기능 외 검증  
  - **인수 테스트**  
    - 알파 테스트: 개발자 참여 하에 내부 테스트  
    - 베타 테스트: 개발자 없이 실제 사용자 환경에서 테스트

- **파레토 법칙 (Pareto Principle)**  
  - 전체 오류의 약 80%가 시스템의 20% 핵심 모듈에서 발생

- **테스트 오라클 (Test Oracle)**  
  - 테스트 결과가 올바른지 판단하기 위한 참값 제공자  
  - 참 오라클: 정확한 결과 제공  
  - 샘플 오라클: 일부 예시 데이터 기준 판단  
  - 휴리스틱 오라클: 경험 기반 판단  
  - 일관성 검사 오라클: 결과 일관성 점검

---

### 추가 설명  
- **살충제 패러독스 (Pesticide Paradox)**는 동일한 테스트 케이스와 동일한 절차만 계속 반복하면 소프트웨어의 새로운 결함을 발견하기 어렵다는 점을 의미합니다.  
- 이는 테스트 케이스를 주기적으로 업데이트하고 다양화하여 새로운 결함을 탐지할 필요가 있음을 시사합니다.

# 문제 4) 1, 2에 해당하는 용어

모듈의 독립성을 판단하는 두가지 지표이다.  
(1)는 모듈과 모듈간의 상호의존 정도,  
(2)는 모듈 내부의 기능적인 집중 정도이다.  

모듈의 독립성을 높이기 위해서는 모듈간의 상호 의존 정도를 나타내는 (1)를 낮추고 모듈이 독립적으로 자체 기능만을 수행하도록 (2)를 높여야 한다.

**답:**  
(1) 결합도  
(2) 응집도

---

## 문제 4 해설

- **재공학**: 이미 있는 것을 개선하거나 재사용하기 위해 고쳐 쓰는 것  
- **역공학**: 소스코드로부터 설계 문서나 명세서 등을 추출하는 작업  
- **재사용**: 잘 만든 모듈이나 컴포넌트를 다시 사용하는 것

- 모듈 설계 시에는  
  - **응집도**를 높이고 (높을수록 좋음)  
  - **결합도**를 낮추는 것(낮을수록 좋음)이 중요함

---

### 응집도 (높을수록 좋음)  
- 우연적 응집: 가장 낮음, 아무 관련 없는 기능을 한 모듈에 모음  
- 논리적 응집: 비슷한 종류의 기능을 모음  
- 시간적 응집: 같은 시간대에 수행되는 기능을 모음  
- 절차적 응집: 순차적인 처리 절차를 모음  
- 통신적 응집: 동일한 입력이나 출력을 사용하는 기능을 모음  
- 순차적 응집: 한 기능의 출력이 다음 기능의 입력으로 사용됨  
- 기능적 응집: 가장 좋음, 단일 기능만 수행

### 결합도 (낮을수록 좋음)  
- 데이터 결합: 가장 낮음, 단순 값만 전달  
- 스탬프 결합: 배열, 구조체 같은 복합 데이터 전달  
- 제어 결합: 제어 정보를 전달  
- 외부 결합: 외부 변수에 의존  
- 공통 결합: 전역 변수를 공유  
- 내용 결합: 가장 높음, 모듈 내부 내용에 직접 접근

---

### 추가 설명  
- 결합도는 모듈 간 의존성을 나타내며 낮을수록 모듈 간 독립성이 커져 유지보수 및 확장성이 좋아진다.  
- 응집도는 모듈 내부 구성 요소들이 얼마나 밀접하게 관련되어 있는지를 나타내며 높을수록 모듈의 응집성이 강해져 이해와 관리가 용이해진다.  
- 좋은 소프트웨어 설계를 위해서는 낮은 결합도와 높은 응집도를 추구해야 한다.

# 문제 5) fan in, fan out 수 구하기

**fan in**과 **fan out**은 소프트웨어 모듈 간의 관계를 나타내는 지표이다.

- **fan in**: 특정 모듈로 들어오는 제어의 수, 즉 몇 개의 모듈이 해당 모듈을 호출하는지를 나타낸다.  
- **fan out**: 특정 모듈에서 나가는 제어의 수, 즉 해당 모듈이 몇 개의 모듈을 호출하는지를 나타낸다.

---

## 답

- **fan in**: 해당 모듈을 호출하는 모듈의 수  
- **fan out**: 해당 모듈이 호출하는 모듈의 수

JSON

# 문제 7) 애플리케이션 성능 측정을 위한 지표

(1): 일정 시간 내에 애플리케이션이 처리하는 일의 양  
(2): 애플리케이션에 요청을 전달한 시간부터 응답이 도착할 때까지 걸린 시간  
(3): 애플리케이션에 요청을 전달한 시간부터 처리가 완료될 때까지 걸린 시간  

자원 사용률: 애플리케이션이 작업을 처리하는 동안의 CPU 사용량, 메모리 사용량, 네트워크 사용량 등 자원 사용률  

---

## 답  
(1) 처리량 (Throughput)  
(2) 응답시간 (Response Time)  
(3) 반환시간 (Turn Around Time)  

---

## 추가 설명  
- **처리량(Throughput)**: 단위 시간당 처리되는 작업의 양으로, 시스템의 처리 성능을 나타낸다.

- **반환시간(Turn Around Time)**: 요청이 완료되어 결과가 반환될 때까지 전체 걸린 시간으로, 배치 작업이나 전체 처리 시간 측정에 사용된다.  
- **자원 사용률**: CPU, 메모리, 네트워크 등 시스템 자원의 사용 정도를 나타내며, 성능 최적화 및 병목 현상 분석에 활용된다.

# 문제 8) 학생(STUDENT)테이블에 전자과 50명, 전기과 100명, 건축과 50명 정보가 저장되어 있을 때 튜플 수를 쓰시오 (DEPT 컬럼은 학과명)

(1) SELECT DEPT FROM STUDENT;  
(2) SELECT DISTINCT DEPT FROM STUDENT;  
(3) SELECT COUNT(DISTINCT DEPT) FROM STUDENT WHERE DEPT = '건축과';

---

## 답  
(1) 200  
(2) 3  
(3) 1  

---

## 추가 설명  
- (1) `SELECT DEPT FROM STUDENT;`는 학생 테이블에서 모든 학과명(DEPT) 컬럼의 데이터를 가져오므로, 전체 학생 수인 200개 튜플이 반환된다.  
- (2) `SELECT DISTINCT DEPT FROM STUDENT;`는 중복을 제거하고 학과명을 한 번씩만 반환하므로, 전자과, 전기과, 건축과 총 3개의 튜플이 반환된다.  
- (3) `SELECT COUNT(DISTINCT DEPT) FROM STUDENT WHERE DEPT = '건축과';`는 건축과로 필터링한 후 유일한 학과명 개수를 세므로, 결과는 1이 된다.

# 문제 9) 네트워크 공격유형

- 출발지 IP와 목적지 IP가 같은 패킷을 만들어 보내는 공격  
- 수신자가 응답을 보낼 때 목적지 주소가 자기 자신이므로 SYN 신호가 계속 자신의 서버를 돌게 되어 서버의 자원을 고갈시켜 가용성을 파괴한다  
- 방화벽에서 출발지와 목적지가 같은 패킷은 모두 제거하여 대응한다  

---

## 답  
Land attack

---

## 문제 9 해설  
서비스 공격 유형의 종류  

**Dos (Denial of Service, 서비스 거부)**: 내가 주체가 되어 공격  

- Smurf Attack: IP, ICMP 프로토콜 이용  
- SYN flooding: SYN 패킷을 과도하게 보내 서버 자원 고갈  
- UDP 플러딩: UDP 패킷 대량 전송  
- Ping 플러딩: Ping 패킷 대량 전송  
- Ping of Death: 규정 크기 이상의 패킷 전송  
- Teardrop Attack: 패킷 재조립 불가 상태 발생  
- Land Attack: 출발지와 목적지가 같은 IP 주소로 패킷을 보내 서버를 공격  

---

# 문제 11) HRN(Highest Response-ratio Next) 우선 순위 계산식

HRN(Highest Response-ratio Next) 우선 순위를 결정하는 계산식을 쓰시오.

---

## 답  
\[
\text{우선순위} = \frac{\text{대기시간} + \text{서비스 시간}}{\text{서비스 시간}}
\]

---

## 문제 11 해설  
- **프로세스 스케줄링**  
  - **선점 스케줄링**: SRT, RR, MLQ(다단계 큐), MFQ(다단계 피드백 큐)  
  - **비선점 스케줄링**: FCFS, SJF, HRN, 기한부, 우선순위  

- **기아 현상**(Starvation): SRT, MLQ, SJF, 우선순위 스케줄링에서 발생 가능  
- **에이징(Aging)**: MFQ, HRN 스케줄링에서 기아 현상을 방지하기 위한 기법  

---

📌 **관련 개념 설명**  
HRN 스케줄링은 대기 시간이 긴 프로세스에게 우선순위를 높게 부여하여, 기아 현상을 완화하는 비선점 방식 스케줄링 알고리즘이다.  
대기시간과 서비스 시간을 합산하여 비율을 계산하고, 이 값이 높은 프로세스를 우선 실행한다.

# 문제 12) 트랜잭션 주요 특성 4가지

(1): 트랜잭션의 가장 기본적인 특성으로 트랜잭션 내에 연산은 반드시 모두 수행되어야 하며, 그렇지 못한 경우 모두 수행되지 않아야 함  
일관성(consistency): 트랜잭션이 정상적으로 완료된 후 언제나 일관성 있는 데이터베이스 상태가 되어야 하며, 결과에 모순이 생겨서는 안 됨  

(2): 하나의 트랜잭션이 수행 중에는 다른 트랜잭션이 접근할 수 없고 각각의 트랜잭션은 독립적이어야 함  

영속성(Durablity): 지속성이라고 하며, 트랜잭션이 성공적으로 완료된 후 결과는 지속적으로 유지되어야 함  

---

## 답  
(1) 원자성(Atomicity)  
(2) 격리성(Isolation)  

---

## 문제 12 해설  
트랜잭션의 주요 특성은 ACID 원칙으로 요약된다.  
- **원자성(Atomicity)**: 트랜잭션의 작업들은 모두 수행되거나 모두 수행되지 않아야 한다.  
- **일관성(Consistency)**: 트랜잭션이 완료된 후 데이터베이스는 일관된 상태를 유지해야 한다.  
- **격리성(Isolation)**: 동시에 수행되는 트랜잭션들은 서로 간섭 없이 독립적으로 처리되어야 한다.  
- **영속성(Durability)**: 트랜잭션이 완료되면 그 결과는 영구적으로 데이터베이스에 반영되어야 한다.  

---

📌 **관련 개념 설명**  
트랜잭션은 데이터베이스의 무결성과 신뢰성을 보장하는 핵심 개념이며, ACID 특성을 통해 데이터 일관성과 안정성을 유지한다.
